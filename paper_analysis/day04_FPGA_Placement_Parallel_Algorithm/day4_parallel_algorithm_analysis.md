## 1. 并行化的核心难题

### 1.1 传统 FPGA 合法化为什么难以并行？

FPGA 合法化需要同时解决两个耦合问题：
- **布局问题**：将单元放到物理位置，最小化线长
- **打包问题**：将单元组合到 CLB Slice 中，满足架构约束（控制集、LUT/FF 数量限制）

传统方法的执行流程是顺序的：

```
传统流程：打包 → 合法化 → 详细布局
  打包：贪心聚类（依赖前序决策）
  合法化：逐行/逐列扫描（依赖邻居状态）
  详细布局：逐单元优化（依赖全局状态）
```

> **核心矛盾**：打包和布局天然耦合——单元应该打包在一起往往是因为它们在物理上靠近，而它们应该放在哪里又取决于它们和谁打包在一起。传统方法将两者分开处理（先打包再布局），不仅损失解质量，而且每一步内部的贪心依赖使得并行化极其困难。

### 1.2 并行化的两大挑战

| 挑战 | 描述 | 传统后果 |
|------|------|---------|
| **数据依赖** | 后续决策依赖前序结果（如"单元 A 放在 Slice X 后，单元 B 不能再去 Slice X"） | 必须顺序执行 |
| **质量退化** | 并行执行可能导致冲突（如两个 Slice 同时竞争同一个单元） | 并行版本解质量差于串行版本 |

> **关键目标**：设计一个既可大规模并行，又保证与串行版本完全等价的算法。

## 2. 大学录取类比：并行性的直觉来源

### 2.1 Gale-Shapley 大学录取问题

论文的核心灵感来自经典的 **Gale-Shapley 大学录取问题（College Admission Problem）**：

| 论文概念 | 大学录取类比 | 含义 |
|---------|------------|------|
| Slice（CLB 切片） | 大学（College） | 有名额限制的接收方 |
| Cell（单元） | 学生（Student） | 需要被接收的请求方 |
| 共享网表的单元 | 朋友（Friends） | 有社交关系的群体 |
| 控制集约束 | 专业限制 | 某些学生不能进同一大学 |

### 2.2 为什么大学录取天然可并行？

大学录取的关键性质：**所有大学可以同时独立做出录取决策**。

```
串行版本：大学 A 做决策 → 大学 B 做决策 → ... → 大学 N 做决策
并行版本：大学 A, B, ..., N 同时做决策 → 学生处理收到的多个 offer → 下一轮
```

```mermaid
graph TD
    A["<b>并行阶段①</b><br/>所有 Slice 同时<br/>生成候选簇"] --> B["<b>并行阶段②</b><br/>所有 Cell 同时<br/>选择最优 offer"]
    B --> C{"<b>还有新 offer？</b>"}
    C -->|"是"| A
    C -->|"否"| D["<b>算法终止</b><br/>合法布局完成"]
```

### 2.3 与经典大学录取的两个关键差异

论文指出，FPGA 合法化比标准大学录取问题更复杂：

1. **学生评价大学不仅看大学本身，还看朋友的决策**：一个单元是否选择某个 Slice，取决于它的"朋友"（同网表的单元）是否也在同一个 Slice 中——这引入了单元间的间接依赖
2. **某些学生不能进入同一大学**：FPGA 架构约束（控制集限制、LUT/FF 数量限制）使得某些单元不能在同一个 Slice 中共存

> 尽管有这些额外复杂度，算法仍然保持了大学录取问题的核心并行结构——每个 Slice 独立决策，每个 Cell 独立选择。

## 3. 节点中心算法：每个 Slice 独立决策

### 3.1 数据结构

每个计算节点（Slice）维护以下数据结构：

| 变量 | 类型 | 含义 |
|------|------|------|
| `det` | 集合 | 已确定属于该 Slice 的单元集合 |
| `pq` | 优先队列 | 当前最优的 K 个候选簇（K=10） |
| `nbr` | 集合 | 邻域内可考虑的候选单元集合 |
| `scl` | 列表 | 种子簇——用于生成新候选 |
| `i` | 整数 | 自 `pq.top()` 上次变化以来的迭代次数 |

### 3.2 节点中心算法流程

每个 Slice 在每次 DL 迭代中执行以下流程：

```mermaid
graph TD
    A["<b>检查 pq.top()</b><br/>是否可提交？"] -->|"是：i ≥ I 且<br/>所有 Cell 接受"| B["<b>提交候选</b><br/>det ← pq.top()<br/>重置 pq, scl"]
    A -->|"否"| C["<b>清理无效项</b><br/>移除无效候选/Cell"]
    C --> D{"<b>邻域太小？</b><br/>|nbr| < N_min"}
    D -->|"是"| E["<b>扩展搜索范围</b><br/>d ← d + Δd<br/>添加更多邻居"]
    D -->|"否"| F["<b>生成新候选</b><br/>将 nbr 中 Cell<br/>加入 scl 中种子簇"]
    E --> F
    F --> G["<b>更新稳定性计数</b><br/>pq.top() 变化？i=0 : i++"]
    G --> H["<b>广播 offer</b><br/>将 pq.top() 发送给<br/>其中所有 Cell"]
```

### 3.3 关键参数与数值

| 参数 | 值 | 含义 |
|------|-----|------|
| K | 10 | 优先队列中保留的最优候选数 |
| I | 3 | 提交前需稳定的迭代次数（防止提交过早的候选） |
| $N_{NBRmin}$ | 10 | 邻域最小 Cell 数 |
| d（初始） | 1 | 初始最大位移约束 |
| Δd | 1 | 每次扩展的位移增量 |
| D | 12 | 最终最大位移约束 |

### 3.4 逐步追踪示例

假设 Slice S1 当前状态：`det = {c1, c2}`，`nbr = {c3, c4, c5, c6}`，`pq.top() = {c1, c2, c3}`：

#### 步骤 1：检查 pq.top() 是否可提交

这是每轮迭代的**第一个判断**，决定当前最优候选是否"成熟"到可以被确认。

**提交条件**——必须**同时**满足两个：

| 条件 | 含义 | 为什么需要 |
|------|------|-----------|
| **i ≥ I（i ≥ 3）** | 稳定性计数器达到阈值 | 防止提交"昙花一现"的候选——只有连续 3 轮迭代中 `pq.top()` 都没变，才说明这是真正稳定的好候选 |
| **所有 Cell 接受** | `pq.top()` 中每个 Cell 都接受了本 Slice 的 offer | 保证不会出现冲突——一个 Cell 只能属于一个 Slice |

**追踪示例：**

1. **检查稳定性**：`pq.top() = {c1, c2, c3}`，当前 `i = 1`，而 `I = 3` → **i < I** ❌
2. **检查 Cell 接受**：Cell c3 还在接受其他 Slice 的 offer（在阶段③中 c3 可能选择了分数更高的 Slice）→ **c3 未接受** ❌
3. **结论**：两个条件任一不满足 → **不可提交**，进入后续步骤

> **为什么需要稳定性条件？** 如果不等稳定就提交，可能会把一个"看上去不错但很快会被更好候选替代"的簇提交了，导致错过更优打包方案，后续需要 rip-up 重做，增加开销。例如：
> ```
> 迭代 1: pq.top() = {c1,c2,c3}  i=1 → 如果此时提交
> 迭代 2: 本可以生成 {c1,c2,c3,c7} SCORE=3.5（更优）
>         但已经提交了 {c1,c2,c3} SCORE=3.2 → 错失更优方案
> ```
>
> **为什么需要所有 Cell 接受？** 这源于 offer-accept 协议（大学录取模型）。如果 c3 在阶段③拒绝了 S1 而选择了 S2，那么 S1 的 `pq.top()` 中包含 c3 就意味着该候选**不可行**，自然不能提交。

**提交后的操作：**

当两个条件都满足时，执行提交：
```
det ← pq.top()    // 将候选簇确认为该 Slice 的最终成员
pq  ← ∅           // 清空优先队列
scl ← {det}       // 将当前确定簇作为新的种子簇
i   ← 0           // 重置稳定性计数器
```

#### 步骤 2：清理无效项

当步骤 1 判定 `pq.top()` 不可提交后，清理所有已经过时/失效的数据。

**清理对象有两类：**

| 清理对象 | 所在数据结构 | 失效原因 |
|---------|------------|---------|
| **已被其他 Slice 确定的 Cell** | `nbr`（邻域候选集） | 该 Cell 已属于别的 Slice，不可能再成为本 Slice 的候选 |
| **包含无效 Cell 的候选簇** | `pq`（优先队列） | 候选簇中包含了已属于其他 Slice 的 Cell，该候选不再可行 |

**追踪示例：**

```
清理前:
  nbr = {c3, c4, c5, c6}
  pq  = [{c1,c2,c3}, {c1,c2,c4}, {c1,c2,c5}, ...]

收到消息：c5 已被 Slice S3 确定（在阶段③中 c5 选择了 S3 的 offer）

清理 nbr:
  c5 已属于 S3 → 从 nbr 中移除
  nbr = {c3, c4, c6}

清理 pq:
  候选 {c1,c2,c5} 包含无效 Cell c5 → 从 pq 中移除
  pq  = [{c1,c2,c3}, {c1,c2,c4}, ...]  （c5 相关的候选全部删除）
```

> **为什么必须清理？** 如果不清理会产生两个严重问题：
>
> 1. **浪费计算资源**：S1 仍然认为 c5 可用，在步骤 4 生成新候选时会继续把 c5 加入种子簇，产生无效候选，纯粹浪费计算。
>
> 2. **阻塞更优候选进入 pq**：pq 容量 K=10，无效候选占了位置，新生成的有效候选可能因分数不够高而被挡在 pq 之外，错过真正可行的更优候选。
>
> 清理所需的信息来自**阶段③**（Cell 并行选择）的反馈——每个 Cell 接受/拒绝 offer 的信息在同步点传回各个 Slice。

#### 步骤 3：检查邻域大小 & 扩展搜索范围

清理完无效 Cell 后，检查邻域是否还"够大"。如果不够，扩大搜索范围。

**触发条件：**

```
if |nbr| < N_min (即 < 10):
    扩展搜索范围
```

**追踪示例：**

```
清理后状态：
  S1.nbr = {c3, c4, c6}     ← 清理掉了 c5
  当前 d = 1

步骤 3 检查：
  |nbr| = 3 < N_min = 10  → 需要扩展！

扩展过程：
  d ← 1 + 1 = 2
  添加距离在 (1, 2] 范围内的 Cell：
    c7 (dist=1.3), c8 (dist=1.5), c9 (dist=1.8),
    c10 (dist=1.2), c11 (dist=1.9), c12 (dist=2.0)

  nbr = {c3, c4, c6, c7, c8, c9, c10, c11, c12}
  |nbr| = 9

  还是不够？→ 如果 |nbr| 仍然 < 10：
    d ← 2 + 1 = 3，继续添加距离在 (2, 3] 的 Cell...
    直到 |nbr| ≥ 10
```

**位移约束 d 的物理意义**——"愿意把 Cell 移多远"：

```
FPGA 芯片布局视图：

         d=1 范围          d=2 范围           d=3 范围
       ┌─────────┐     ┌───────────┐     ┌─────────────┐
       │ c3  c4  │     │ c7  c8    │     │ c13  c14    │
       │    S1   │     │    c9     │     │    c15      │
       │ c6      │     │ c10 c11  │     │    c16      │
       └─────────┘     │    c12    │     └─────────────┘
        4个Cell        └───────────┘       更多Cell...
                       +6个Cell

     移动代价小         移动代价中等         移动代价大
     打包质量高         打包质量中等         打包质量可能较差
```

- **d 小** → 只考虑近邻 Cell → 位移代价低（HPWL 增量小）→ 但候选少，可能找不到好的打包
- **d 大** → 考虑更远的 Cell → 候选更多 → 但位移代价高，评分函数中 $-\lambda \cdot \text{HPWL}$ 的惩罚增大

> **为什么要渐进式扩展，而不是一开始就用最大范围 d=D？**
>
> | 策略 | 优点 | 缺点 |
> |------|------|------|
> | **渐进式** (d=1→2→3→...) | 优先尝试低位移代价的候选，节省计算量 | 可能需要多轮扩展 |
> | **一次性** (d=D=12) | 一次看到所有候选 | 计算量暴增；远处 Cell 的候选大量涌入，可能淹没近处的好候选 |
>
> 渐进式扩展确保了：如果近邻中就有好的打包方案，直接找到，无需看更远的；只有在近处不够时才逐步放宽。这是**先尝试代价小的方案，不够再逐步放宽**的核心原则。

**扩展后可能的情况：**

- 情况 A：`|nbr| ≥ N_min` → 邻域足够大，进入步骤 4
- 情况 B：即使 `d=D=12`，`|nbr|` 仍然 `< N_min` → 该 Slice 附近确实没有足够的 Cell，本轮迭代中无法生成有效候选，等待后续迭代

#### 步骤 4：生成新候选

这是节点中心算法中**最核心的步骤**——基于当前邻域中的 Cell 和已有的种子簇，生成所有可能的新候选簇，评估分数，保留最优的 K 个。

**生成方式：将邻域 Cell 加入种子簇**

```
对于 nbr 中每个 Cell c：
  对于 scl 中每个种子簇 seed：
    生成新候选 = seed ∪ {c}
    如果新候选合法（满足架构约束）：
      计算分数，尝试加入 pq
```

**追踪示例：**

```
当前状态：
  S1.det = {c1, c2}
  S1.nbr = {c3, c4, c6, c7, c8, c9, c10, c11, c12}
  S1.scl = { {c1, c2} }     ← 种子簇
  S1.pq = [{c1, c2, c3}]    ← 优先队列（清理后剩余）

生成过程（种子簇 {c1, c2}）：
  c3 加入 → {c1, c2, c3}    ← 已在 pq 中
  c4 加入 → {c1, c2, c4}
  c6 加入 → {c1, c2, c6}
  c7 加入 → {c1, c2, c7}
  c8 加入 → {c1, c2, c8}
  ...
```

如果 `scl` 中有多个种子簇：

```
scl = { {c1, c2}, {c1, c2, c3} }

  种子 {c1, c2}：
    + c4 → {c1, c2, c4}
    + c7 → {c1, c2, c7}
    ...

  种子 {c1, c2, c3}：
    + c4 → {c1, c2, c3, c4}
    + c7 → {c1, c2, c3, c7}
    ...
```

> **关键**：种子簇越大，生成的新候选也越大。算法不仅探索"加入一个 Cell"的方案，还探索"在已有较大簇上继续叠加"的方案。

**合法性检查**——不是所有候选都能进 pq，必须通过 FPGA 架构约束：

| 约束 | 含义 | 示例 |
|------|------|------|
| **LUT 数量限制** | 一个 Slice 中 LUT 数 ≤ 8（UltraScale） | {c1,...,c9} 含 9 个 LUT → 非法 ❌ |
| **FF 数量限制** | 一个 Slice 中 FF 数 ≤ 16（UltraScale） | 类似 |
| **控制集约束** | 同一 Slice 内的 FF 必须共享同一控制集（时钟、复位等） | c3 的时钟 = clk_A，c7 的时钟 = clk_B → 不兼容 ❌ |

```
示例：{c1, c2, c3, c7} 的合法性检查
  LUT 数 = 4 ≤ 8   ✅
  FF 数  = 6 ≤ 16  ✅
  控制集兼容性:
    c1: {clk_A, rst}
    c2: {clk_A, rst}
    c3: {clk_A, rst}
    c7: {clk_B, rst}    ← 不同时钟！❌
  → 控制集不兼容，该候选非法，丢弃
```

**计算评分**——合法候选用评分函数评估（详见 Section 6）：

$$\text{SCORE}(c, s) = \underbrace{\sum_{e \in \text{Net}(c)} \frac{\text{InternalPins}(e, c)}{\text{TotalPins}(e) - 1}}_{\text{打包质量项}} - \underbrace{\lambda \cdot \text{HPWL}(c, s)}_{\text{位移惩罚项}}$$

```
计算 {c1, c2, c7} 的分数:
  打包质量 = 3.5
  c7 距离 S1 = 1.3 → HPWL 增量 = 35 → 惩罚 = 0.02 × 35 = 0.7
  SCORE = 3.5 - 0.7 = 2.8

计算 {c1, c2, c4} 的分数:
  打包质量 = 2.0
  c4 距离 S1 = 0.8 → HPWL 增量 = 20 → 惩罚 = 0.02 × 20 = 0.4
  SCORE = 2.0 - 0.4 = 1.6
```

**保留 Top-K**——pq 最多保留 K=10 个最优候选，新候选需要和已有候选竞争：

```
当前 pq（按分数降序）:
  1. {c1,c2,c3}    SCORE = 3.2
  ...
  10. {c1,c2,c12}  SCORE = 1.0   ← pq 中最低分

新候选 {c1,c2,c10} SCORE = 0.8 → 比 pq 最低分还低 → 丢弃
新候选 {c1,c2,c3,c7} SCORE = 2.6（如控制集兼容）→ 挤掉最低分 → 进入 pq
```

> **为什么 K=10？** K 太小（如 K=1）：一旦唯一候选被 Cell 拒绝，没有备选方案，算法效率低。K 太大（如 K=1000）：维护优先队列开销大，且 top-10 之后的候选分数极低，被选中的概率极小，浪费资源。K=10 在备选方案数量和维护开销之间取得了平衡。

**种子簇 scl 的演化**——种子簇决定了"从什么基础上继续添加 Cell"：

```
迭代 0（初始化）:
  scl = { ∅ }                ← 空簇，从零开始

迭代 1:
  生成 {c1}, {c2}, {c3}, ...
  假设 pq.top() 稳定后提交了 {c1, c2}
  scl = { {c1, c2} }        ← 提交后，det 成为新种子

迭代 2:
  从 {c1, c2} 基础上继续添加
  生成 {c1,c2,c3}, {c1,c2,c7}, {c1,c2,c3,c7}, ...
  假设提交了 {c1, c2, c3, c7}
  scl = { {c1, c2, c3, c7} }

迭代 3:
  从 {c1, c2, c3, c7} 基础上继续添加
  直到无法再添加（Slice 已满或无合法候选）
```

> 这体现了**渐进式构建**策略：每一轮迭代在已有成果上逐步扩展，而不是每次从零开始重新搜索。同时，**只枚举"加一个 Cell"的增量操作**，而不是枚举所有可能的子集——这使得候选数量从指数级（$2^{|nbr|}$）降低到线性级（$|scl| \times |nbr|$），同时通过多轮迭代逐步收敛到最优方案。

#### 步骤 5：更新稳定性计数器

这是对算法收敛至关重要的步骤——它决定了 `pq.top()` 何时"成熟"到可以被提交。

**规则只有两条：**

```
if pq.top() 发生了变化:
    i ← 0       ← 发现了更好的候选，重置计数器
else:
    i ← i + 1   ← 当前最优没变，稳定性 +1
```

**追踪示例——多轮迭代中 i 的变化：**

```
迭代 1:
  pq.top() = {c1,c2,c3}

迭代 2:
  步骤 4 生成了新候选 {c1,c2,c3,c7}: 3.5 > 3.2
  → pq.top() 变成了 {c1,c2,c3,c7}
  步骤 5: pq.top() 变了！  i ← 0    ← 重置，重新开始计数

迭代 3:
  步骤 4 没有生成比 {c1,c2,c3,c7} 更好的候选
  pq.top() 仍然是 {c1,c2,c3,c7}
  步骤 5: pq.top() 没变    i ← 0+1 = 1

迭代 4:
  pq.top() 仍然没变
  步骤 5: pq.top() 没变    i ← 1+1 = 2

迭代 5:
  pq.top() 仍然没变
  步骤 5: pq.top() 没变    i ← 2+1 = 3
  → 回到步骤 1：i=3 ≥ I=3，稳定性条件满足！
```

**pq.top() 变化的三种来源：**

| 来源 | 说明 | 意味着 |
|------|------|--------|
| 步骤 4 生成了更好的新候选 | 新候选分数更高，挤入 pq 顶部 | 搜索尚未收敛，当前最优会被取代 → 重置 i ✅ |
| 步骤 2 清理了 pq.top() 中的无效 Cell | pq.top() 被移除，次优候选浮到顶部 | 当前最优"不靠谱"，需要等新候选稳定 → 重置 i ✅ |
| 阶段③ Cell 拒绝导致候选失效 | Cell 拒绝了 offer，候选不可行 | 同上，下一轮清理时 pq.top() 会变 → 重置 i ✅ |

> 来源 2 和 3 都意味着当前最优候选"不靠谱"——它包含了不可能获得的 Cell。此时重置 i 是完全正确的：需要等一个新的、真正可行的候选稳定下来。

**稳定性计数器的状态机视角：**

```
                    pq.top() 变化
                 ┌──────────────────┐
                 │                  │
                 ▼                  │
  ┌─────┐   i=0   ┌─────┐   i=1   ┌─────┐   i=2   ┌─────┐   i=3
  │重置 │ ──────→ │计数1│ ──────→ │计数2│ ──────→ │计数3│ ──────→ 可提交！
  └─────┘         └─────┘         └─────┘         └─────┘
     ▲               │               │               │
     │  pq.top()变化  │  pq.top()变化 │  pq.top()变化
     └───────────────┴───────────────┴───────────────┘
                      任何时刻变化都回到 i=0
```

**为什么 I=3？**

| I 的值 | 行为 | 问题 |
|--------|------|------|
| I=1 | pq.top() 只要一轮不变就提交 | 太急躁，可能刚巧这一轮没生成更好的候选，但下一轮就会有。过早提交导致错失更优方案 |
| I=3 | 连续 3 轮不变才提交 | 平衡点。3 轮不变说明邻域中确实没有更好的选择了，此时提交合理 |
| I=10 | 连续 10 轮不变才提交 | 太保守，浪费迭代轮次，延迟收敛。3 轮不变已经是很强的信号 |

> 稳定性计数器承担了三重角色：
> 1. **质量守门员**：防止过早提交次优候选
> 2. **效率调节器**：避免无意义的频繁提交和撤回
> 3. **收敛保证器**：与确定性平局打破一起，确保算法无振荡地收敛（防止"提交→撤回→提交→撤回"的振荡）

#### 步骤 6：广播 offer

- 将 `pq.top()` 中的 Cell 列表和评分发送给每个 Cell
- Cell 们将在并行阶段③中做出选择

#### 步骤间的闭环关系

步骤 5 和步骤 1 构成一个**闭环**，步骤 1-6 形成完整的迭代循环：

```
步骤 1: 读取 i，判断是否可提交
         ↓ (不可提交)
步骤 2: 清理无效项（可能导致 pq.top() 变化）
         ↓
步骤 3: 检查邻域大小，必要时扩展搜索范围
         ↓
步骤 4: 生成新候选（可能导致 pq.top() 变化）
         ↓
步骤 5: 更新 i（根据 pq.top() 是否变化）
         ↓
步骤 6: 广播 pq.top() 的 offer 给其中所有 Cell
         ↓
阶段③: Cell 并行选择
         ↓
回到步骤 1: 读取更新后的 i，再次判断
```

## 4. 完整并行 DL 算法：三阶段并行结构

### 4.1 Algorithm 1 完整伪代码

```
输入：粗糙合法布局，Cell 集合 C，Slice 集合 S，初始位移约束 d
输出：合法布局

═══════════════════════════════════════════
阶段①：并行初始化
═══════════════════════════════════════════
parallel for each s ∈ S do
  s.i   ← 0
  s.det ← ∅
  s.pq  ← ∅
  s.nbr ← {c ∈ C | dist(c, s) ≤ d}
  s.scl ← {空簇}
end parallel for

═══════════════════════════════════════════
主循环：迭代直到无新 offer
═══════════════════════════════════════════
while ∃ valid pq.top() for s ∈ S do

  ──── 阶段②：Slice 并行决策 ────
  parallel for each s ∈ S do
    执行节点中心算法（图 7）
  end parallel for

  ──── 阶段③：Cell 并行选择 ────
  parallel for each c ∈ C do
    从 {s ∈ S | c ∈ s.pq.top()} 中
    选择 SCORE(s) 最高的 Slice s'
    并告知 s' 接受其 offer
  end parallel for

end while
```

### 4.2 三阶段并行结构详解

```mermaid
graph LR
    subgraph Phase1["阶段① 初始化"]
        S1["Slice 1<br/>init"]
        S2["Slice 2<br/>init"]
        SN["Slice N<br/>init"]
    end
    subgraph Phase2["阶段② Slice 决策"]
        P1["Slice 1<br/>生成 offer"]
        P2["Slice 2<br/>生成 offer"]
        PN["Slice N<br/>生成 offer"]
    end
    subgraph Phase3["阶段③ Cell 选择"]
        C1["Cell 1<br/>选最优 offer"]
        C2["Cell 2<br/>选最优 offer"]
        CM["Cell M<br/>选最优 offer"]
    end
    Phase1 --> Phase2 --> Phase3
    Phase3 -->|"下一轮迭代"| Phase2
```

| 阶段 | 并行对象 | 操作 | 同步点 |
|------|---------|------|--------|
| ① 初始化 | 所有 Slice | 设置初始状态 | 全部完成后进入主循环 |
| ② Slice 决策 | 所有 Slice | 独立执行节点中心算法，生成 offer | 全部完成后进入阶段③ |
| ③ Cell 选择 | 所有 Cell | 处理收到的多个 offer，选择最优 | 全部完成后回到阶段② |

> **同步模型**：算法采用 **BSP（Bulk Synchronous Parallel）** 模型——每个阶段内部完全并行，阶段之间通过全局同步点分隔。这种模型非常适合多线程 CPU（OpenMP）和 GPU（CUDA）实现。

### 4.3 为什么这个结构可以并行？

**阶段②可以并行的原因**：
- 每个 Slice 只读取自己的 `det`, `pq`, `nbr`, `scl` 状态
- 每个 Slice 只写入自己的 `pq`（优先队列）和广播消息
- 不同 Slice 的写入不冲突——它们操作不同的数据结构

**阶段③可以并行的原因**：
- 每个 Cell 只需要读取来自各 Slice 的 offer（只读）
- 每个 Cell 独立做出选择，写入自己的决定
- 不同 Cell 的写入不冲突

**阶段之间的数据流**：
- 阶段② → 阶段③：Slice 广播 offer（多对多通信）
- 阶段③ → 阶段②：Cell 通知接受/拒绝（多对多通信）

> **无竞争条件**：阶段②中 Slice 不读取其他 Slice 的状态，阶段③中 Cell 不修改其他 Cell 的决定。唯一的数据交换通过"offer/accept"消息完成，在同步点处统一处理。

## 5. 串行等价性：并行不损失质量

### 5.1 定义

> **串行等价性（Serial Equivalency）**：一个并行算法满足串行等价性，当且仅当它总是产生与串行版本**完全相同**的解——无论使用多少线程。

### 5.2 为什么串行等价性重要？

| 性质 | 有串行等价性 | 无串行等价性 |
|------|------------|------------|
| 结果可重复 | ✅ 每次运行结果相同 | ❌ 结果随线程数变化 |
| 调试友好 | ✅ 可用单线程版本验证 | ❌ 并行/串行结果不同 |
| 线程数灵活 | ✅ 有多少用多少 | ⚠️ 可能多线程更差 |
| 理论保证 | ✅ 等价于良定义的串行算法 | ❌ 无理论保证 |

### 5.3 串行等价性的保证机制

**Lemma 1**：假设 $s_1$ 和 $s_2$ 是两个不同的计算节点。如果对同一 DL 迭代中的任意 $s_1$ 和 $s_2$，$\text{SCORE}(s_1) \neq \text{SCORE}(s_2)$ 成立，则算法收敛性得到保证。

**实现方式**：当多个 Slice 对同一个 Cell 提供相同的分数改善时，使用 Slice 的**唯一标识符**打破平局。这保证了：
1. 每个 Cell 在每次迭代中做出**唯一确定**的选择
2. 串行版本和并行版本中，Cell 面临的 offer 集合相同
3. Cell 的选择逻辑相同（选分数最高的，平局按 ID）
4. 因此最终结果完全一致

> **与 ABCDPlace 的对比**：ABCDPlace（Day 5）的并行全局交换算法需要顺序应用交换结果（ApplyCand 步骤）来处理冲突，因此**不保证**串行等价性。本文的 DL 算法通过"offer-accept"协议和确定性平局打破，**保证**了串行等价性。

### 5.4 收敛性分析

算法的收敛性由以下机制保证：

1. **分数单调递增**：每次成功提交（commit）都严格改善目标函数值
2. **有限解空间**：Cell-Slice 的组合是有限的
3. **无振荡**：确定性平局打破避免了"A→B→A→B"的振荡

因此算法在有限步内收敛。

## 6. 评分函数：布局与打包的统一度量

### 6.1 评分函数定义

给定一个 Slice $s$ 和一个候选簇 $c$，评分函数定义为：

$$\text{SCORE}(c, s) = \sum_{e \in \text{Net}(c)} \frac{\text{InternalPins}(e, c)}{\text{TotalPins}(e) - 1} - \lambda \cdot \text{HPWL}(c, s)$$

其中：
- $\text{Net}(c)$：至少包含 $c$ 中一个 Cell 的网集合
- $\text{TotalPins}(e)$：网 $e$ 的总引脚数
- $\text{InternalPins}(e, c)$：网 $e$ 在簇 $c$ 内的引脚数
- $\text{HPWL}(c, s)$：将 $c$ 中 Cell 从 FIP 位置移到 Slice $s$ 造成的 HPWL 增量
- $\lambda = 0.02$：权重参数

### 6.2 两项的物理意义

**第一项（打包质量项）**：

$$\sum_{e \in \text{Net}(c)} \frac{\text{InternalPins}(e, c)}{\text{TotalPins}(e) - 1}$$

> **解读**：这项衡量簇的"内部化"程度——将外部网变为内部网的能力。当更多引脚被吸收到簇内部时，该簇需要的对外路由资源减少，布线更友好。分母 $\text{TotalPins}(e) - 1$ 使得吸收大网（引脚数多的网）的收益相对较低——因为大网即使部分内部化，仍需对外连接。

**第二项（布局位移项）**：

$$-\lambda \cdot \text{HPWL}(c, s)$$

> **解读**：这项惩罚位移造成的线长增加。负号意味着 HPWL 增加越多的候选得分越低。$\lambda$ 控制"打包质量"与"线长"之间的权衡。

### 6.3 数值计算示例

假设簇 $c = \{c_1, c_2, c_3\}$，候选 Slice $s$：

| 网 $e$ | TotalPins | InternalPins | 贡献 |
|--------|-----------|-------------|------|
| $e_1$ (连接 $c_1, c_2, x_1$) | 3 | 2 | $2/(3-1) = 1.0$ |
| $e_2$ (连接 $c_2, c_3$) | 2 | 2 | $2/(2-1) = 2.0$ |
| $e_3$ (连接 $c_1, x_2, x_3$) | 3 | 1 | $1/(3-1) = 0.5$ |

打包质量项 = $1.0 + 2.0 + 0.5 = 3.5$

假设移动到 Slice $s$ 的 HPWL 增量为 50：

$$\text{SCORE} = 3.5 - 0.02 \times 50 = 3.5 - 1.0 = 2.5$$

> 直觉上：$e_2$ 完全内部化（贡献最高 2.0），$e_1$ 大部分内部化（贡献 1.0），$e_3$ 仅部分内部化（贡献 0.5）。该簇的打包质量较好，且位移代价适中。

### 6.4 分数改善量

Cell 选择 Slice 时使用的不是绝对分数，而是**分数改善量**：

$$\text{SCORE}(s) = \text{SCORE}(\text{pq.top()} \text{ of } s, s) - \text{SCORE}(\text{det} \text{ of } s, s)$$

> 即：如果接受 Slice $s$ 的 offer，$s$ 的分数会比当前状态改善多少。Cell 自然倾向于选择改善最大的 Slice。

## 7. 后 DL 异常处理：撕毁与重分配

### 7.1 问题

DL 主循环结束后，通常有 < 1% 的 Cell 仍无法在位移约束 D 内找到合法位置。

### 7.2 撕毁评分函数

对于非法 Cell $v$ 和 Slice $s$（其已确定簇为 $c$），撕毁 $c$ 以容纳 $v$ 的评分：

$$\text{SCORE}_{ripup}(v, s, c) = -\lambda_1 \cdot \text{HPWL}(v, s) - \lambda_2 \cdot \text{SCORE}(c, s) - \lambda_3 \cdot \text{Area}(c)$$

参数：$\lambda_1 = 0.02$，$\lambda_2 = 1.0$，$\lambda_3 = 4.0$

> **三项的含义**：
> - $-\lambda_1 \cdot \text{HPWL}$：优先选择移动 $v$ 线长代价小的 Slice
> - $-\lambda_2 \cdot \text{SCORE}$：优先撕毁当前分数低的簇（即质量较差的簇）
> - $-\lambda_3 \cdot \text{Area}$：优先撕毁面积小的簇——面积大的簇要么包含很多 Cell（重新合法化困难），要么包含难以打包的 Cell

### 7.3 撕毁-重分配流程（Algorithm 2）

```
for each 非法 Cell c:
  D(c) ← D  (初始位移约束)
  while Legalize(c, D(c)) 失败:
    D(c) ← D(c) + 1  (逐步放宽约束)

function Legalize(c, D):
  ls(c) ← {s ∈ S | dist(c, s) ≤ D}  (候选 Slice)
  按 SCOREripup 降序排列 ls(c)
  for each s in ls(c):
    if RipUpAndLegalize(s, c, D) 成功:
      return 成功
  return 失败

function RipUpAndLegalize(s, c, D):
  lv ← {v ∈ C | v ∈ s.det}  (记录将被撕毁的 Cell)
  s.det ← {c}  (将非法 Cell c 放入 s)
  for each v in lv:  (逐个重新合法化被撕毁的 Cell)
    ls(v) ← {s' ∈ S | dist(v, s') ≤ D 且 s'.det ∪ {v} 合法}
    if ls(v) 为空:
      恢复 s.det 原状，return 失败
    else:
      选择使分数改善最大的 s'
      s'.det ← s'.det ∪ {v}
  return 成功
```

> **注意**：后 DL 异常处理是**顺序执行**的——因为撕毁一个簇后重新分配其 Cell 需要逐步检查，每一步都依赖前一步的结果。但由于涉及的 Cell 数极少（< 1%），顺序执行的开销很小。

| **冲突处理** | 顺序 ApplyCand，放弃冲突 | 确定性平局打破 |
| **串行等价** | ❌ 不保证 | ✅ 保证 |
| **粒度** | 粗粒度 batch（256 个单元/批） | 细粒度（每个 Slice/Cell 一个线程） |
| **理论基础** | 工程实践驱动 | Gale-Shapley 理论驱动 |

> **核心差异**：ABCDPlace 采用"乐观并行+冲突回退"策略——先并行执行，遇到冲突就放弃。本文采用"协议并行+确定性选择"策略——通过 offer-accept 协议避免冲突，通过确定性平局打破保证串行等价。后者更优雅但应用场景更受限（需要问题本身具有"大学录取"结构）。

## 10. 并行算法设计方法论总结

### 10.1 从本文提炼的并行算法设计模式

```mermaid
graph TD
    A["<b>顺序算法</b><br/>本质贪心/依赖"] --> B{"<b>能否找到<br/>并行类比？</b>"}
    B -->|"能"| C["<b>设计 offer-accept 协议</b><br/>每方独立决策<br/>通过消息通信"]
    B -->|"不能"| D["<b>设计批量执行</b><br/>放宽依赖约束<br/>冲突时回退"]
    C --> E["<b>保证串行等价？</b>"]
    E -->|"能"| F["<b>✅ 理想模式</b><br/>可重复、可验证<br/>线程数无关"]
    E -->|"不能"| G["<b>加入确定性机制</b><br/>平局打破/优先级"]
    D --> H["<b>❌ 近似模式</b><br/>可能损失质量<br/>需实验验证"]
```

### 10.2 Offer-Accept 协议的设计要点

1. **分离决策与通信**：每个节点独立决策（生成 offer），通过消息传递协调
2. **确定性选择**：当存在多个等价 offer 时，使用确定性规则打破平局
3. **BSP 同步**：阶段之间全局同步，阶段内部完全并行
4. **单调改善**：每轮迭代目标函数单调改善，保证收敛
5. **有限状态**：解空间有限，算法在有限步内终止

### 10.3 适用条件

Offer-Accept 协议适用于以下场景：
- 问题可建模为**双边匹配**（two-sided matching）
- 两方可以**独立评估**对方（评分函数可分解）
- 存在**自然的多对多通信**模式（每个 Slice 可能涉及多个 Cell，反之亦然）
- 可以容忍**迭代收敛**（非一步到位）

> **不适用的场景**：强耦合的优化问题（如连续优化中的梯度下降）、全局最优依赖（如整数规划的分支定界）、需要严格同步的流水线操作。
